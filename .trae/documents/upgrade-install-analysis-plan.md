# 新版本升级安装方案分析与改造计划

## Summary

- 目标：分析当前项目的新版本升级安装方案是否存在缺陷，给出可复用的通用升级安装方案，并形成一份适用于本仓库的渐进式改造计划。
- 当前项目是一个 `macOS-only` 的 Electron 应用，已接入 `electron-updater + GitHub Releases`，具备“检查更新、下载更新、重启安装”的基础能力，但整体仍停留在“可用的最小闭环”，缺少发布链路、签名公证、版本通道、失败兜底和面向 AI Agent 的可执行发布契约等关键能力。
- 本计划优先输出“分析 + 改造计划”，并按“长期通用方案”视角设计，但会基于当前仓库的真实结构和现有文件给出可直接落地的改造步骤。
- 额外约束：考虑到实际发布操作主要由 AI Agent 执行，方案将优先强调“脚本化、结构化、可校验、幂等、少人工判断”，而不是依赖人工阅读长篇操作手册。

## Current State Analysis

### 已确认的现状

- `electron/main.ts`
  - 已使用 `electron-updater`。
  - 已开启启动自动检查：`autoUpdater.checkForUpdatesAndNotify()`。
  - 已提供手动检查、下载、安装 IPC：`update:check` / `update:download` / `update:install`。
  - 已监听 `update-available`、`update-not-available`、`download-progress`、`update-downloaded`、`error` 事件。
  - 当前配置为 `autoDownload = false`、`autoInstallOnAppQuit = true`、`allowPrerelease = true`。
- `electron-builder.json`
  - 当前仅配置 `mac.target = ["dmg", "zip"]`。
  - 发布源为 GitHub Releases：`provider = github`。
  - 未看到签名、公证、架构策略、artifact 命名、channel 策略等增强配置。
- `README.md`
  - 已说明从 GitHub Release 下载最新版本。
  - 已说明发布步骤：手动改 `package.json` 版本、打 tag、执行 `npm run build:mac`、手动执行 `gh release create` 上传 `dmg`、`zip`、`latest-mac.yml`。
- 仓库中未发现 `.github/workflows/*`，说明当前没有自动化发布流水线。
- 仓库中未发现显式的 notarization / codesign 配置文件或说明。

### 当前方案的主要缺陷

1. 分发闭环可用，但发布链路对 AI Agent 不友好且不可靠
- 现在的发布主要体现在 README 的命令示例中，缺少单一入口脚本、预检脚本和结构化校验结果。
- 即使发布由 AI Agent 执行，也仍然容易出现漏传 `latest-mac.yml`、漏传 zip、tag 与 version 不一致、Release 标题/说明不统一等问题。
- 对自动更新来说，客户端能否更新取决于发布产物是否完整；只要缺少机器可校验的发布契约，Agent 执行也会有线上失败风险。

2. 缺少签名与公证策略，macOS 安装/升级体验存在系统层风险
- 当前仓库未体现签名、公证、证书注入、Apple 凭据配置等内容。
- 对 macOS 应用而言，未签名/未公证会直接影响首次安装、覆盖安装、自动更新、Gatekeeper 信任和用户安全感知。

3. 预发布策略过于宽松，稳定用户可能收到不应接收的版本
- `electron/main.ts` 中 `allowPrerelease = true` 是全局启用。
- 在没有稳定版 / beta / alpha 通道治理时，稳定用户有机会看到预发布版本，破坏版本预期。

4. 启动即检查的用户体验与错误兜底不完整
- 启动自动检查是合理的，但当前 `update-not-available` 会直接发出“当前已是最新版本”提示，容易造成每次启动都弹出成功类提示，形成噪音。
- `error` 事件仅 `console.error`，没有用户可见提示、错误分类、重试建议或日志落地。

5. 缺少版本治理与升级兼容设计
- 当前方案只覆盖“拿到新包并安装”，但没有体现升级前后数据兼容、配置迁移、最低支持版本、回滚策略。
- 这在桌面应用迭代到中后期后会成为主要故障来源之一。

6. 缺少面向 AI Agent 的发布契约与面向用户的安装兜底说明
- README 虽说明可以从 Release 下载，但没有把“自动更新失败时如何回退到手动安装”“如何判断是更新源问题还是客户端问题”“如何验证发布产物完整性”沉淀成机器优先、人工可读的稳定流程。
- 对 AI Agent 而言，当前缺少明确的输入输出约定，例如发布前检查项、必需产物、失败退出条件、成功判定标准。

7. 缺少自动化验证
- 仓库中未看到用于验证更新产物的脚本或 CI。
- 当前没有自动校验 `dmg`、`zip`、`latest-mac.yml`、版本号、tag、Release 资产一致性的机制。

8. 通用性不足
- 当前方案与 GitHub Releases、macOS、Electron 的默认路径强绑定。
- 若未来扩展到多平台、多通道、私有分发源或企业内网分发，现有结构缺少抽象层和发布规范，不利于演进。

## 通用方案

### 推荐的通用升级安装方案（适用于桌面应用）

1. 版本治理
- 明确定义 `stable / beta / alpha` 通道。
- 约束版本语义：采用 semver，并规定预发布版本命名规则。
- 客户端依据通道决定是否接收预发布版本，而不是全量放开。

2. 构建与签名
- 为每个平台产出标准安装包和更新包。
- 所有可分发产物必须经过签名；macOS 需要公证。
- 构建过程由 CI 驱动，避免人工本地构建成为唯一可信来源。

3. 发布与元数据
- 发布时同时上传安装包、增量/更新包、更新元数据、校验信息和发布说明。
- 发布产物必须是不可变资产；客户端依赖元数据做版本发现与校验。
- Release 创建、资产上传、完整性校验应自动化。

4. AI Agent 友好的发布契约
- 所有发布动作都应收敛到少量确定命令，例如“构建”“校验”“发布”三个稳定入口。
- 每个入口都应有明确退出码、结构化输出和失败原因，便于 Agent 判断是否继续。
- 发布前检查、产物校验、Release 资产一致性必须脚本化，而不是依赖 Agent 自己理解 README。

5. 客户端升级状态机
- 启动静默检查，不打扰用户。
- 发现新版本后展示版本号、变更摘要、包大小与下载动作。
- 下载进度可感知；下载完成后支持“立即重启安装”与“稍后安装”。
- 检查失败、下载失败、校验失败、安装失败分别处理，并提供明确兜底。

6. 手动安装兜底
- 始终保留“从官方下载最新安装包覆盖安装”的路径。
- 自动更新只是首选路径，不是唯一升级路径。
- 在客户端 UI 和项目文档中明确“自动更新失败时如何处理”。

7. 兼容与迁移
- 升级前检查最低支持版本与关键环境约束。
- 如涉及本地配置、缓存、数据库或任务调度，必须有版本化迁移逻辑。
- 迁移失败应可回滚或进入安全降级模式。

8. 观测与回滚
- 记录升级链路的关键事件：检查、发现、下载、安装、失败原因。
- 支持快速下架错误 Release、暂停某通道、回退到上一个稳定版本。

### 对当前项目的通用落点

- 短期继续使用 `electron-updater + GitHub Releases` 完全可行。
- 中期应补齐：签名/公证、版本通道、自动化发布、失败兜底、发布校验。
- 长期如需扩展，可抽象“更新提供方”和“版本通道策略”，让客户端不再直接依赖单一发布平台的默认行为。

## Proposed Changes

### 一、先把当前 GitHub Releases 方案补齐为“可靠且适合 AI Agent 执行”

#### 1. 调整客户端升级策略

- 文件：`electron/main.ts`
- 变更内容：
  - 将更新逻辑从入口文件内联代码整理为更清晰的更新状态机或独立模块。
  - 区分“启动静默检查”和“用户手动检查”两种场景。
  - 仅在手动检查时展示“当前已是最新版本”，启动自动检查时不主动 toast。
  - 为 `error`、下载失败、安装失败提供用户可见提示和可操作兜底。
  - 将 `allowPrerelease = true` 改为依据通道配置决定，而不是全局放开。
- 原因：
  - 降低提示噪音。
  - 防止稳定用户误收预发布版本。
  - 让失败可理解、可恢复。
- 实施方式：
  - 新增更新场景标识，例如 `silent` / `manual`。
  - 在主进程维护当前检查来源和下载状态。
  - 统一封装更新事件到 UI 通知层。

#### 2. 强化打包与发布配置

- 文件：`electron-builder.json`
- 变更内容：
  - 明确产物命名规则和目标架构策略。
  - 评估是否改为生成 `universal` 包，或分别产出 `arm64` 与 `x64` 并在文档中说明。
  - 增加与签名、公证、发布 channel 兼容的配置占位。
- 原因：
  - 当前只有最基础的 `dmg + zip` 输出配置，缺少面向稳定发布的结构化约束。
- 实施方式：
  - 补充 artifact 命名、架构说明、发布元数据规范。
  - 为后续 CI 与签名公证接入留下固定入口。

#### 3. 固化发布脚本与校验脚本

- 文件：`package.json`
- 新增文件：`scripts/release/verify-mac-release.mjs`
- 变更内容：
  - 增加标准化脚本，如 `release:build:mac`、`release:verify:mac`、`release:publish:mac`。
  - 校验 `package.json.version`、git tag、输出文件名、`latest-mac.yml` 是否存在且版本一致。
  - 输出结构化结果，至少让 AI Agent 能稳定判断 `preflight passed`、`artifacts complete`、`publish ready`。
- 原因：
  - 降低 Agent 发布出错概率，减少对自然语言说明的依赖。
- 实施方式：
  - 用脚本替代 README 中零散命令。
  - 后续 CI 直接复用同一脚本，避免“双份流程”。

#### 4. 补齐 AI Agent 发布契约与用户兜底说明

- 文件：`README.md`
- 新增文件：`docs/release-process.md`
- 可选新增文件：`docs/release-contract.md`
- 变更内容：
  - 把“自动更新”和“手动覆盖安装”的区别写清楚。
  - 把发布流程改写为“给 AI Agent 执行的固定命令 + 成功判定标准 + 失败退出条件”。
  - 说明何时上传 `dmg`、`zip`、`latest-mac.yml`，以及三者缺一不可。
  - 仅保留必要的人类说明，不再把 README 作为主要执行入口。
- 原因：
  - 让升级安装流程从“看文档执行”变成“按契约执行并可被 Agent 稳定复用”。

### 二、补齐 macOS 正式分发必须能力

#### 5. 增加签名与公证方案

- 文件：`electron-builder.json`
- 新增文件：`.github/workflows/release-mac.yml`
- 可选新增文件：`scripts/release/notarize-readme.md`
- 变更内容：
  - 在计划实施阶段接入 Apple Developer 签名与 notarization。
  - 通过 CI 注入证书与 Apple 凭据，不在本地仓库存明文。
- 原因：
  - 这是 macOS 正式安装与自动更新可靠性的前置条件之一。
- 实施方式：
  - 在 CI 中完成构建、签名、公证、上传 Release 资产。
  - 保持本地仍可开发构建，但正式发布只认 CI 产物。

#### 6. 引入自动化发布流水线

- 新增文件：`.github/workflows/release-mac.yml`
- 变更内容：
  - 在 tag 推送时自动构建、签名、公证、校验、创建或更新 Release。
  - 自动上传 `dmg`、`zip`、`latest-mac.yml`。
  - 自动失败终止，避免不完整 Release 暴露给客户端。
- 原因：
  - 自动更新系统对发布资产完整性极度敏感，CI 是最小可信机制。

### 三、为长期通用方案留出演进空间

#### 7. 加入版本通道与升级策略抽象

- 文件：`electron/main.ts`
- 可选新增文件：`electron/services/updater.ts`
- 可选新增文件：`src/types/update.ts`
- 变更内容：
  - 将通道策略抽象为配置，例如 `stable`、`beta`。
  - UI 可展示当前通道与接收策略。
  - 更新逻辑不直接散落在主进程入口。
- 原因：
  - 为未来多通道、灰度、私有源或企业内部分发做准备。

#### 8. 为升级失败与兼容迁移预留机制

- 文件：`electron/main.ts`
- 可选新增文件：`electron/services/migrations.ts`
- 变更内容：
  - 在版本升级后执行必要的本地配置迁移。
  - 记录升级前版本与迁移结果。
  - 出现不兼容时提供安全降级或引导用户手动处理。
- 原因：
  - 桌面应用的“升级安装方案”不应只包含下载与覆盖，也要覆盖升级后的运行正确性。

## Assumptions & Decisions

- 决策：当前项目继续以 `GitHub Releases + electron-updater` 为第一阶段基础方案，不在第一轮就替换更新基础设施。
- 决策：改造目标以 `macOS` 为主，因为仓库当前仅配置了 mac 产物，没有发现其他平台实际实现。
- 决策：优先把“稳定发布能力”补齐，再考虑多平台、多源、灰度分发等扩展能力。
- 决策：后续实施以“分析 + 改造计划”为主，不默认引入重型平台服务。
- 决策：发布流程以 AI Agent 为主要执行者，因此优先建设脚本入口、结构化校验和 CI，而不是增加人工步骤。
- 假设：项目未来仍会持续发版，因此值得投入 CI、签名、公证和发布校验。
- 假设：当前用户群体至少包含稳定版用户，因此不应继续全局开放预发布版本接收。

## Verification Steps

### 计划阶段验证

- 确认以下现状已被计划覆盖：
  - `electron/main.ts` 中的更新策略、提示时机和错误处理。
  - `electron-builder.json` 中的打包与发布配置增强。
  - `package.json` 中的标准化发布脚本。
  - `README.md` 与新增文档中的 AI Agent 发布契约、发布/兜底流程说明。
  - `.github/workflows/release-mac.yml` 中的自动化发布设计。

### 实施阶段验收标准

- 手动检查更新时：
  - 有新版能看到版本信息并进入下载。
  - 无新版时才提示“当前已是最新版本”。
- 启动自动检查时：
  - 无新版不打扰用户。
  - 有新版时能清晰提示并进入下载流程。
- 发布阶段：
  - AI Agent 只需要调用固定脚本或 CI 入口即可完成发布，不需要自行拼接临时命令。
  - 脚本输出能够明确表示预检通过、产物完整、可发布或失败原因。
  - CI 能自动生成并上传 `dmg`、`zip`、`latest-mac.yml`。
  - 版本号、tag、Release 资产名称一致。
  - 缺少关键资产时发布失败而不是产出不完整 Release。
- 安装阶段：
  - 用户可以通过自动更新升级。
  - 自动更新失败时，可以通过 Release 页面下载最新安装包覆盖安装。
- 长期演进：
  - 版本通道策略可配置。
  - 预发布不会默认影响稳定用户。
